# ZYLCH PERSONA ARCHITECTURE EVOLUTION

## MISSION CONTEXT

Zylch is a research lab building relational memory infrastructure. The thesis: LLMs are perfect for relational intelligence because professional relationships are already encoded in language. The missing piece is persistent relational memory - that's what Zylch builds.

**Core concept: AVATAR** - A vector-based representation of a person, built from interaction history across all channels (email, calendar, WhatsApp, phone calls).

**Key principles:**
- A person is NOT an email address. One person can have multiple identifiers (emails, phones, names)
- Memory reconsolidation: Similar memories UPDATE (not duplicate), mimicking human memory
- Small-world topology for retrieval
- Shareable avatars: When employees leave, relational knowledge stays

## CURRENT STATE (WHAT EXISTS)

### Architecture (from ZYLCH_MEMORY_ARCHITECTURE.md):
- ZylchMemory: Vector-based semantic storage with HNSW indexing (O(log n) retrieval)
- Sentence-transformers embeddings (384-dim, all-MiniLM-L6-v2)
- Supabase pg_vector for storage
- Namespace hierarchy: global → user → contact
- Avatar structure defined but NOT fully implemented
- Confidence tracking with Bayesian updates
- Memory lifecycle management (TTL, decay, garbage collection)

### Current Problem:
The `/tasks` command (formerly `/gaps`) is making Anthropic API calls for EACH contact to categorize emails into task summaries. This is:
1. WRONG - Zylch's mission is building relational memory, NOT email categorization
2. EXPENSIVE - N API calls per contact
3. SLOW - Sequential analysis

The analysis should come FROM the avatar/memory system, not from per-request LLM calls.

## DESIRED STATE

### Phase 1: Data Ingestion (No LLM calls needed)
When emails sync:
1. Extract interaction data (who, when, what channel, direction in/out)
2. Store in vector database per-contact namespace
3. Build/update avatar incrementally from raw data

### Phase 2: Avatar Building (Background, async)
Build avatar profile from accumulated data:
- Communication patterns (frequency, response latency, preferred times)
- Relationship strength (recency, frequency weighted)
- Interaction history summary
- NO per-request LLM calls - avatar is PRE-COMPUTED

### Phase 3: Query Time (Fast retrieval)
When user asks "show me tasks" or "info on Luigi":
1. Semantic search in vector store
2. Retrieve pre-computed avatar
3. Format and display - NO LLM call needed for basic queries
4. Optional: LLM only for complex reasoning/synthesis
